# 团队模式模板

## 模式一：功能开发团队

**适用场景**：新功能实现、跨层变更（前端+后端+测试）

**典型结构**：
| 角色 | agent 类型 | 职责 | 文件范围 |
|------|-----------|------|---------|
| 领导 | 当前会话 | 规划、协调、合并 | 不直接编码 |
| 架构师 | Plan | 设计 API 契约、数据模型 | 只读分析 |
| 前端实现 | general-purpose | UI 组件、状态管理 | src/components/, src/pages/ |
| 后端实现 | general-purpose | 路由、控制器、存储 | src/api/, src/services/ |
| 测试专员 | general-purpose | 单元/集成/e2e 测试 | tests/ |

**关键规则**：
- 架构师先完成设计，其他人再开始实现
- 前后端各自独立分支，避免文件冲突
- 测试专员等实现完成后再写测试

**任务依赖示例**：
```
架构设计 (task-1) ← 前端实现 (task-2) blockedBy [task-1]
                  ← 后端实现 (task-3) blockedBy [task-1]
                  ← 测试编写 (task-4) blockedBy [task-2, task-3]
```

## 模式二：代码审计团队

**适用场景**：全面代码审查、PR Review、安全审计

**典型结构**：
| 角色 | agent 类型 | 职责 |
|------|-----------|------|
| 领导 | 当前会话 | 汇总报告、解决分歧 |
| 安全专家 | Explore | 认证漏洞、注入风险、密钥泄露 |
| 性能专家 | Explore | 打包体积、查询优化、渲染性能 |
| 测试覆盖 | Explore | 测试完整性、边界情况、覆盖率 |

**关键规则**：
- 所有审查者并行工作
- 每人输出结构化报告（评分 + 具体文件:行号）
- 领导合成最终报告

## 模式三：调试/假设竞争团队

**适用场景**：复杂 bug 排查、根因分析

**典型结构**：
| 角色 | agent 类型 | 职责 |
|------|-----------|------|
| 领导 | 当前会话 | 分配假设、裁决结论 |
| 假设 A | Explore | 调查假设 A（如：内存泄漏） |
| 假设 B | Explore | 调查假设 B（如：竞态条件） |
| 假设 C | Explore | 调查假设 C（如：配置错误） |

**关键规则**：
- 每个假设独立调查
- 鼓励互相质疑（通过消息）
- 找到证据后向领导报告

## 模式四：研究辩论团队

**适用场景**：技术选型、架构决策、方案评估

**典型结构**：
| 角色 | agent 类型 | 职责 |
|------|-----------|------|
| 领导 | 当前会话 | 最终决策 |
| 正方 | general-purpose | 全力支持方案 A |
| 反方 | general-purpose | 全力反对，提出替代方案 |
| 研究员 | Explore | 搜索真实案例、benchmark |
| 合成者 | general-purpose | 总结双方、提出折中 |

**关键规则**：
- 正反方并行辩论
- 研究员提供事实依据
- 合成者在辩论后总结

## 模式五：重构/迁移团队

**适用场景**：大规模重构、架构迁移、模块拆分

**典型结构**：
| 角色 | agent 类型 | 职责 | 文件范围 |
|------|-----------|------|---------|
| 领导 | 当前会话 | 规划、合并 | 不直接编码 |
| 规划师 | Plan | 分析依赖、拆分边界 | 只读 |
| 模块 A | general-purpose | 重构模块 A | module-a/ |
| 模块 B | general-purpose | 重构模块 B | module-b/ |
| 验证者 | general-purpose | 回归测试、对比验证 | tests/ |

**关键规则**：
- 规划师先完成分析
- 每个模块用独立 worktree
- 共享依赖变更需广播讨论

## 模式选择决策树

```
任务类型？
├── 新功能/跨层变更 → 模式一（功能开发）
├── 代码审查/安全审计 → 模式二（代码审计）
├── Bug 排查/根因分析 → 模式三（调试竞争）
├── 技术选型/方案评估 → 模式四（研究辩论）
└── 大规模重构/迁移 → 模式五（重构迁移）
```

## 团队规模建议

| 任务复杂度 | 建议人数 | 说明 |
|-----------|---------|------|
| 简单（单模块） | 2-3 | 1 实现 + 1 测试/审查 |
| 中等（跨模块） | 3-4 | 按模块分工 |
| 复杂（跨层） | 4-6 | 按层/职责分工 |
| 大型（架构级） | 5-8 | 按子系统分工 |

**原则**：宁少勿多。每增加一个 teammate 都增加协调成本和 token 消耗。
