# 多电脑 Git 合并机制说明

## 核心原则

Git 是**按行逐文件合并**，不是理解内容后"求同存异"。以下是对用户（4台电脑：家里电脑、办公室电脑、笔记本、江敏笔记本）的实际影响。

## 各类场景结果

| 场景 | 结果 | 说明 |
|------|------|------|
| 不同电脑新增不同文件 | 自动融合 | 如：A新增技能X，B新增技能Y，两个都在 |
| 同文件不同段落修改 | 自动合并 | 两份内容拼接在一起 |
| 同文件同段/同行内容修改 | 冲突 | 后推送者报错，需手动选保留哪个 |
| 一方改了文件，另一方没动 | 直接覆盖 | 以后推送者内容为准 |
| skills/下新增不同名文件 | 自动融合 | 各自新增，文件名不同就不冲突 |

## 经常改的文件（可能冲突）

| 文件 | 风险 |
|------|------|
| SOUL.md / SOUL_Pro.md / SOUL_Edu.md | 高——两边都改同段角色描述会冲突 |
| config.yaml | 中——每台配置不同，建议一台上改完推，另一台拉取后各自适配 |
| memories/MEMORY.md | 低——Hermes自动追加内容，通常追加在末尾互不冲突 |
| memories/USER.md | 低——同上 |
| skills/ | 低——文件名不同就不会冲突 |

## 冲突解决步骤

当推送报错出现以下信息：

```
Auto-merging memories/MEMORY.md
CONFLICT (content): Merge conflict in memories/MEMORY.md
Automatic merge failed; fix conflicts and then commit the result.
```

处理方式：

1. 打开冲突文件，找到 `<<<<<<<` 和 `>>>>>>>` 标记
2. 删除不需要保留的那段内容
3. 删除标记行（`<<<<<<<`、`=======`、`>>>>>>>`）
4. `git add` 该文件
5. `git commit` 完成合并

## 最佳实践

```
改之前先拉（git pull）
推之前确认本地是最新的
同一台电脑负责改同一类文件，减少冲突概率
```
