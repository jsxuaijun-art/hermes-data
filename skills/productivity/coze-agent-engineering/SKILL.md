# COZE 智能体工程（财税领域）

## 适用场景

用户在扣子（COZE）平台开发多个财税/AI智能体，需要对已有智能体进行：
- **分析**：拆解现有System Prompt的结构、能力、缺失环节
- **整合**：合并多个版本的优点，消除冲突和重叠
- **增强**：加入用户个人的专业背书和差异化卖点
- **迭代**：基于测试反馈持续优化

## 一、前置步骤：读取智能体设计文档

用户经常将智能体的设计思路、迭代历程保存在 Word 文档（.docx）中。

### 1.1 文件定位

COZE 设计文档通常放在 OneDrive/桌面等位置。常见路径：

```
/mnt/d/OneDrive/Desktop/<文件名>.docx
/mnt/c/Users/<用户名>/Desktop/<文件名>.docx
/mnt/c/Users/<用户名>/OneDrive/Desktop/<文件名>.docx
```

搜索命令参考：`find /mnt -name "*<关键词>*" -type f 2>/dev/null`

### 1.2 ⚠️ 关键：docx文本提取的兜底方案

`python-docx` 的 `doc.paragraphs` 在某些复杂格式的docx中可能返回**空结果**（内容在XML层面存在但python-docx没有正确解析）。

**遇到这种情况，不要卡住，切换到底层兜底方案：**

```python
import zipfile, re

z = zipfile.ZipFile('/path/to/file.docx')
xml = z.read('word/document.xml').decode('utf-8', errors='replace')
texts = re.findall(r'<w:t[^>]*>([^<]+)</w:t>', xml)
full_text = ''.join(texts)
print(full_text)
```

**原理**：docx本质是ZIP压缩包，`word/document.xml` 是核心内容文件，`<w:t>` 标签包含所有文本片段。正则提取后拼接即为完整文本。

**不写文件，直接运行**（推荐，避免临时文件残留）：

```bash
python3 -c "
import zipfile, re
z = zipfile.ZipFile('/path/to/file.docx')
xml = z.read('word/document.xml').decode('utf-8')
texts = re.findall(r'<w:t[^>]*>([^<]+)</w:t>', xml)
print(''.join(texts))
"
```

**如果仍然取不到内容**：检查文档是否包含表格（`.tables`）、文本框、SmartArt等特殊元素。对复杂的docx，考虑读取 `word/document.xml` 全文查看XML结构后针对性提取。

## 二、差距分析框架

拿到多个版本的智能体设计后，用以下维度做结构化对比：

| 维度 | 智能体A | 智能体B | 结论 |
|------|---------|---------|------|
| 信息收集完整性 | 30项/50项 | 简单 | A胜 |
| 法规引用规范性 | 无文件号 | 列明文件号 | B胜 |
| 行业研究能力 | 有 | 无 | A胜 |
| 关键词智能路由 | 无 | 有 | B胜 |
| 合规检查独立模块 | 无 | 有 | B胜 |
| 健壮性（跳过机制） | 有 | 无 | A胜 |
| 流程简洁度 | 偏长 | 简洁 | B胜 |
| 老板语言亲和度 | 一般 | 好 | B胜 |

**核心问题**：从对比中发现"一个全面但笨重，一个灵活但单薄"——整合方向是**用A的广度嫁接B的锐度**，同时补上双方都没有的差异化能力。

## 三、整合方法论

### 3.1 合并同类项

两版都有的能力（如"税务筹划方案生成""对话讨论迭代"），保留最强的那个版本，不要简单拼贴。

**规则**：
- 重复的功能 → 保留逻辑更完整、输出结构更清晰的那个
- 冲突的规则 → 优先保留"对用户更友好"的那个（老板视角 > 技术视角）
- 互补的功能 → 同时保留，设定触发条件避免打架

### 3.2 补空白（双方都缺失的）

基于用户的实际业务特征（非通用能力），识别三大类缺失：

1. **专业信任背书** — 高级会计师、行业内唯一、16年经验等
2. **业务场景覆盖** — 企业生命周期（注册→注销）、双城政策等
3. **安全与边界** — 免责声明、违法红线、不确定时的诚实应答

### 3.3 架构重组

新智能体采用**模块化架构**，每个模块有独立触发条件，避免一次性输出全部流程：

```
欢迎引擎 → 智能路由 → 信息收集 → [行业研究 / 法规采集]
                                  → [税务筹划 / 合规检查 / 财报分析]
                                  → 对话迭代 → 定稿
```

每个模块的出口统一设计："是否需要进入XXX模块？"——用户跳过后不影响核心流程运行。

## 四、财税领域增强项模板

基于江姐（高级会计师，20年经验）的用户画像，以下是在智能体中可**自然植入**的差异化能力：

### 4.1 企业生命周期导航

| 阶段 | 用户触发词 | 智能体输出 |
|------|-----------|-----------|
| 初创期 | 注册、初创、新公司 | 注册类型税负对比、核定征收vs查账征收 |
| 成长期 | 扩张、融资、开分店 | 高新认定、研发加计扣除、架构调整 |
| 成熟期 | 利润高、分红、持股 | 利润分配优化、个税筹划、合规升级 |
| 收缩期 | 亏损、生意不好 | 亏损弥补、资产处置节税、裁员个税 |
| 退出期 | 注销、清算、转让 | 清税全流程、股权vs资产转让税负对比 |

### 4.2 双城政策适配

识别企业注册地（苏州/上海/两地都涉及），自动适配：
- 社保基数与比例差异
- 地方税费减免口径
- 产业补贴政策
- 高新认定地方标准

### 4.3 信任背书植入规则

只在以下场景自然带出，不硬塞：
- 用户质疑方案可行性 → "我服务了1000+家苏州/上海的企业..."
- 用户问到高级会计师 → 自然讲述评审经历
- 方案开头可低调带一句："基于20年财税行业经验（含高级会计师资质）"

## 五、System Prompt 输出规范

输出的System Prompt需满足：

1. **文件格式**：纯文本 .txt，直接可粘贴到COZE
2. **字数控制**：12000-15000字（COZE的System Prompt窗口容量合理范围）
3. **模块化分隔**：用 `# 模块名称` 做清晰分区，方便用户后续修改
4. **每个模块包含**：触发条件 + 输出结构模板 + 健壮性规则
5. **Markdown内嵌**：报告输出模板用代码块包裹（\`\`\`markdown ... \`\`\`），防止被COZE渲染打乱

## 六、测试建议

交付后建议用户做A/B对比测试：

```
场景：一家苏州跨境电商，年营收500万，进项发票缺，想节税
1. 用旧版本prompt试一次
2. 用新prompt试一次
3. 对比输出质量差异
```

## 七、已知陷阱

- **不要两个路由系统共存** — COZE平台的内置路由与System Prompt中的关键词路由可能冲突，需明确优先级
- **不要超过COZE上下文窗口** — System Prompt过长时，后续对话中模型可能"遗忘"后面的规则
- **不要使用COZE不支持的格式** — 如复杂的XML/JSON引用、特定函数调用等
- **信息收集不要一次性全部抛出** — 分阶段、分轮次，防止用户被50个问题吓退
